## 0. 문제 정의 및 본질 분석 (Problem Definition & Root Cause) *(필수)*

<!--
  헌법 제1조(Problem Space First)에 따라, 해결책을 제시하기 전에 문제의 본질을 정의합니다.

**문제의 본질**: 사용자는 자신의 아이디어나 문제 상황을 모호하게 인지하고 있으며, 이를 개발 가능한 수준의 구체적인 요구사항이나 문제 기술서로 변환하는 데 어려움을 겪습니다. 초기 단계에서 문제의 본질(Root Cause)이 명확히 규명되지 않으면, 잘못된 해결책을 설계하게 되어 자원이 낭비되는 근본적인 문제가 발생합니다.

**분석**: 
- 사용자는 증상(Symptom)과 원인(Cause)을 혼동하는 경향이 있음.
- 논리적 비약이나 숨겨진 전제를 스스로 인지하기 어려움.
- 체계적인 질문(5 Whys 등) 없이는 심층적인 문제 공간(Problem Space) 탐색이 불가능함.

## 사용자 시나리오 및 테스트 (User Scenarios & Testing) *(필수)*

### 사용자 스토리 1 - 문제 입력 및 진단 시작 (우선순위: P1)

사용자가 모호한 문제 상황(예: "우리 팀의 생산성이 너무 낮은 것 같아요")을 입력하면, AI 에이전트는 사용자의 답변에서 핵심 페르소나와 배경을 추출하고 공감 어린 태도로 첫 번째 심층 질문을 던집니다.

**우선순위 선정 이유**: 시스템의 진입점이자 진단 프로세스의 시작으로, 핵심 가치인 문제 탐색의 기초가 됩니다.

**독립적 테스트 (Independent Test)**: 모호한 문장을 입력했을 때 AI가 텍스트에서 메타데이터를 추출하고 첫 번째 "Why" 질문을 정상적으로 출력하는지 확인합니다.

**수락 시나리오 (Acceptance Scenarios)**:

1. **Given** 사용자가 메인 화면에서 모호한 문제를 입력할 때, **When** '분석 시작'을 누르면, **Then** 시스템은 대화 상태(GraphState)를 초기화하고 첫 번째 질문을 제시해야 한다.
2. **Given** 대화가 시작되면, **When** 시스템이 첫 번째 답변을 분석하면, **Then** 답변에 포함된 사용자 페르소나와 환경적 배경 정보가 메타데이터로 추출되어야 한다.

---

### 사용자 스토리 2 - 5 Whys 심층 문답 진행 (우선순위: P1)

사용자와 AI 에이전트는 최대 5회에 걸쳐 문답을 주고받습니다. AI는 이전 답변의 논리적 허점을 찌르거나 숨겨진 전제를 파악하여 인과관계를 깊게 파고드는 질문을 생성합니다.

**우선순위 선정 이유**: 문제의 근본 원인을 파악하기 위한 핵심 로직(Questioner Node)이 작동하는 구간입니다.

**독립적 테스트 (Independent Test)**: 3~5회의 문답 과정에서 질문의 깊이가 점진적으로 깊어지며, 인과관계가 논리적으로 연결되는지 확인합니다.

**수락 시나리오 (Acceptance Scenarios)**:

1. **Given** AI가 질문을 생성할 때, **When** 이전 답변에 논리적 비약이 있다면, **Then** 해당 비약을 확인하기 위한 질문을 우선적으로 생성해야 한다.
2. **Given** 질문 단계가 진행됨에 따라, **When** GraphState의 질문 단계(Step)가 1에서 5까지 순차적으로 증가해야 한다.

---

### 사용자 스토리 3 - 진단 종료 및 문제 기술서 생성 (우선순위: P1)

근본 원인이 파악되었거나 질문이 5단계에 도달하면 대화가 자동 종료됩니다. 시스템은 수집된 모든 정보를 구조화하여 Markdown 및 JSON 형식의 '문제 기술서'를 생성하여 사용자에게 제공합니다.

**우선순위 선정 이유**: 진단 결과물을 사용자에게 전달하는 최종 단계로, 후속 개발 단계의 입력값이 됩니다.

**독립적 테스트 (Independent Test)**: 대화 종료 후 생성된 파일에 표면적 문제, 인과관계 연쇄, 최종 근본 원인이 모두 포함되어 있는지 검증합니다.

**수락 시나리오 (Acceptance Scenarios)**:

1. **Given** 질문이 5단계에 도달하거나 근본 원인이 파악되면, **When** 엣지 로직(Edge Logic)이 작동하면, **Then** 대화가 즉시 종료되고 결과 생성 단계로 전이되어야 한다.
2. **Given** 결과물이 생성되면, **When** 파일 내용을 확인하면, **Then** '표면적 문제'부터 '최종 근본 원인'까지의 논리적 흐름이 포함되어야 한다.

## 요구사항 (Requirements) *(필수)*

### 기능적 요구사항 (Functional Requirements)

- **FR-001**: **Analyzer Node**: 사용자의 텍스트 입력에서 논리적 비약(Logical Leap)과 숨겨진 전제(Hidden Assumptions)를 식별해야 함.
- **FR-002**: **Questioner Node**: 분석된 결과와 대화 이력을 바탕으로 인과관계를 파고드는 5 Whys 질문을 생성해야 함.
- **FR-003**: **State Management**: LangGraph를 사용하여 대화 이력, 질문 단계(1~5), 메타데이터를 포함하는 GraphState의 무결성을 유지해야 하며, PostgreSQL을 통해 모든 상태를 실시간으로 영구 저장하여 대화 중단 시 재개가 가능해야 함.
- **FR-004**: **Edge Logic**: 근본 원인이 발견되었다고 판단되거나 질문이 5단계에 도달하면 대화를 즉시 종료해야 함.
- **FR-005**: **Result Generation**: 대화 종료 시 모든 상세 데이터를 포함하는 JSON(Full Spec)과 가독성을 고려한 Markdown(요약 보고서)을 동시에 생성해야 함.
- **FR-006**: **한국어 전용**: 시스템은 한국어 입력을 처리하고 한국어로 된 문제 기술서를 생성해야 함. (헌법의 한국어 최우선 원칙 준수)

### 비기능적 요구사항 (Non-Functional Requirements)

- **NFR-001**: **신뢰성**: LLM API 호출 실패 시 지수 백오프 재시도 전략을 적용해야 함.
- **NFR-002**: **유지보수성**: 모든 내부 클래스 및 노드 로직에 한국어 Google Style Docstring을 작성해야 함.
- **NFR-003**: **성능**: 각 질문 생성 단계의 응답 시간은 사용자 경험을 위해 적정 수준(예: < 5초)을 유지해야 함.
- **NFR-004**: **보안**: 사용자의 입력 데이터에 포함된 민감 정보가 로그에 노출되지 않도록 필터링해야 함.

### 주요 엔티티 (Key Entities)

- **GraphState**: 대화의 전체 상태(이력, 단계, 추출된 메타데이터)를 담는 데이터 구조.
- **ProblemSpecification**: 최종 결과물인 문제 기술서 엔티티 (표면적 문제, 인과 연쇄, 전제, 근본 원인 포함).
- **Metadata**: 사용자의 페르소나, 배경 지식, 전문성 수준 등을 담는 정보.

## 성공 기준 (Success Criteria) *(필수)*

### 측정 가능한 결과 (Measurable Outcomes)

- **SC-001**: 모든 대화는 예외 없이 5단계 이내에 논리적으로 완결되어야 함.
- **SC-002**: 생성된 문제 기술서의 '인과관계 연쇄'가 최소 3단계 이상의 논리적 연결을 포함해야 함.
- **SC-003**: 사용자 만족도 조사에서 "생성된 질문이 문제의 핵심을 찌르는가?"라는 항목에 80% 이상의 긍정 응답률 확보.
- **SC-004**: 기술서의 JSON 데이터가 사전에 정의된 Schema 유효성 검사를 100% 통과해야 함.
