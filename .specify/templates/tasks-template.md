# 작업 목록 (Tasks): [FEATURE NAME]

**입력 (Input)**: `/specs/[###-feature-name]/`에서 가져온 설계 문서
**선행 조건 (Prerequisites)**: plan.md (필수), spec.md (사용자 스토리용 필수), research.md, data-model.md, contracts/

**테스트 (Tests)**: 아래 예시에는 테스트 작업이 포함되어 있습니다. 테스트는 선택 사항(OPTIONAL)입니다. 기능 명세서에서 명시적으로 요청된 경우에만 포함하십시오. 모든 테스트는 Django 프레임워크 권장 방식을 따릅니다.

**조직 (Organization)**: 각 스토리를 독립적으로 구현하고 테스트할 수 있도록 사용자 스토리별로 작업이 그룹화됩니다.

## 형식: `[ID] [P?] [Story] 설명`

- **[P]**: 병렬 실행 가능 (서로 다른 파일, 의존성 없음)
- **[Story]**: 이 작업이 속한 사용자 스토리 (예: US1, US2, US3)
- 설명에 정확한 파일 경로를 포함하십시오

## 경로 관례 (Path Conventions)

- **Django 프로젝트**: `src/config/`, `src/apps/[app_name]/`, `src/core/`, `tests/`
- 아래에 표시된 경로는 Django 표준 레이아웃을 가정합니다 - plan.md 구조에 따라 조정하십시오

## 의존성 및 실행 순서 (Dependencies & Execution Order)

### 단계별 의존성 (Phase Dependencies)

- **설정 (Phase 1)**: 의존성 없음 - 즉시 시작 가능
- **기반 작업 (Phase 2)**: 설정 완료에 의존 - 모든 사용자 스토리를 차단함
- **사용자 스토리 (Phase 3+)**: 모두 기반 작업 단계 완료에 의존
  - 사용자 스토리는 병렬로 진행될 수 있음 (인력이 충분한 경우)
  - 또는 우선순위 순서(P1 → P2 → P3)에 따라 순차적으로 진행됨
- **마무리 (최종 단계)**: 모든 원하는 사용자 스토리가 완료되는 것에 의존

### 사용자 스토리 의존성 (User Story Dependencies)

- **사용자 스토리 1 (P1)**: 기반 작업(Phase 2) 후에 시작 가능 - 다른 스토리에 대한 의존성 없음
- **사용자 스토리 2 (P2)**: 기반 작업(Phase 2) 후에 시작 가능 - US1과 통합될 수 있지만 독립적으로 테스트 가능해야 함
- **사용자 스토리 3 (P3)**: 기반 작업(Phase 2) 후에 시작 가능 - US1/US2와 통합될 수 있지만 독립적으로 테스트 가능해야 함

### 각 사용자 스토리 내부에서

- 테스트(포함된 경우)는 반드시 구현 전에 작성되고 실패해야 함
- 모델은 서비스 전에 작성
- 서비스는 뷰(View) 및 엔드포인트 전에 작성
- 핵심 구현은 통합 전에 작성
- 다음 우선순위로 이동하기 전에 스토리가 완료되어야 함

### 병렬 실행 기회 (Parallel Opportunities)

- [P] 표시가 된 모든 설정 작업은 병렬로 실행 가능
- [P] 표시가 된 모든 기반 작업은 병렬로 실행 가능 (Phase 2 내부에서)
- 기반 작업 단계가 완료되면, 모든 사용자 스토리를 병렬로 시작할 수 있음 (팀 역량이 허용하는 경우)
- [P] 표시가 된 사용자 스토리의 모든 테스트는 병렬로 실행 가능
- [P] 표시가 된 스토리 내부의 모델들은 병렬로 실행 가능

## 구현 전략 (Implementation Strategy)

### MVP 우선 (사용자 스토리 1만)

1. Phase 1: 설정 완료
2. Phase 2: 기반 작업 완료 (중요 - 모든 스토리 차단)
3. Phase 3: 사용자 스토리 1 완료
4. **중단 및 검증**: 사용자 스토리 1을 독립적으로 테스트
5. 준비된 경우 배포/시연

### 점진적 인도 (Incremental Delivery)

1. 설정 + 기반 작업 완료 → 기반 준비 완료
2. 사용자 스토리 1 추가 → 독립적으로 테스트 → 배포/시연 (MVP!)
3. 사용자 스토리 2 추가 → 독립적으로 테스트 → 배포/시연
4. 사용자 스토리 3 추가 → 독립적으로 테스트 → 배포/시연
5. 각 스토리는 이전 스토리를 깨뜨리지 않고 가치를 추가함

### 병렬 팀 전략

여러 개발자가 있는 경우:

1. 팀이 함께 설정 + 기반 작업을 완료함
2. 기반 작업이 완료되면:
   - 개발자 A: 사용자 스토리 1
   - 개발자 B: 사용자 스토리 2
   - 개발자 C: 사용자 스토리 3
3. 스토리들이 독립적으로 완료되고 통합됨
